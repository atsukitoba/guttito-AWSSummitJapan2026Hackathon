# User Stories

## Overview

gutitto の要件（`aidlc-docs/inception/requirements/requirements.md`）を、プライマリペルソナ「めぐみ」（話して整理する人）の**ユーザージャーニー**に沿った INVEST ストーリーに変換する。

### ストーリー構成

- **粒度**: 中粒度（全15本）
- **エピック**: ユーザージャーニー軸（起動前 / 通話中 / 通話後）＋ 横断
- **形式**: 「〜として、〜したい。なぜなら〜だから。」
- **受け入れ基準**: Given/When/Then 形式
- **メタ情報**: 優先度（Must/Should/Could）＋ 3本柱タグ
- **要件トレーサビリティ**: 対応する `R-*` 要件IDを明記
- **審査訴求ポイント**: 主要ストーリーのみ1行明記

### 3本柱タグ凡例

- 🫂 **受容** = 受け止める力（R-Accept）
- ⚡ **ゼロUX** = R-Entry / NFR-Zero-UX
- 🧠 **YouKnowMe** = あなたを知っていること（R-YouKnowMe）
- 🔄 **横断** = 複数の柱・共通基盤

### 優先度凡例

- **Must** = 予選 MVP 必須
- **Should** = 予選デモで見せたい
- **Could** = 決勝フェーズで実装

## エピック1: 起動前 — 呼ばれなくてもそこに居る

めぐみが gutitto に触れる前から、gutitto は彼女のことを知っている。ウィジェットが視界の隅にあり、1タップで繋がる準備ができている。

---

### Story 1.1: ロック画面から1タップで話し始める

**【ストーリー】**
プライマリペルソナ めぐみ として、ベッドに寝転がってスマホを触る気力すらない夜でも、**ロック画面のウィジェットをタップするだけで通話を始めたい**。なぜなら、画面ロック解除・アプリ起動・通話ボタンタップの3ステップが、疲れた夜には "大きすぎるハードル" になるから。

**【タグ】** ⚡ ゼロUX | **【優先度】** Must | **【対応要件】** R-Entry-1, R-Entry-2, NFR-Zero-UX-1, NFR-Zero-UX-2, NFR-Performance-2

**【受け入れ基準】**

```
シナリオ1: ロック画面ウィジェットから1タップ起動
Given: スマホがロック状態で、ロック画面にgutittoウィジェットが配置されている
When: ユーザーがウィジェットをタップする
Then: 画面ロックを解除せず、NFR-Performance-2（3秒以内）で音声セッションが開始される
  And: ペルソナから先に挨拶が始まる

シナリオ2: ホーム画面からの1タップ起動
Given: スマホがロック解除された状態でホーム画面にいる
When: ユーザーが gutitto アプリアイコンをタップする
Then: NFR-Performance-2（3秒以内）で通話画面が表示される
  And: 通話画面には巨大な通話ボタン1つと、ペルソナの顔しか表示されていない
  And: 通話ボタンをタップすると音声セッションが開始される

シナリオ3: 通話画面に余計な選択肢がない
Given: 通話画面が表示されている
When: ユーザーが画面を見る
Then: メニュー・入力欄・選択肢・同意ダイアログは一切表示されていない
  And: ペルソナの顔と通話ボタンだけが見える

シナリオ4: ロック画面からの起動時間（遅延目標）
Given: ロック画面ウィジェットからの起動
When: ウィジェットをタップする
Then: 起動から通話画面表示まで NFR-Performance-2（3秒以内）で完了する
  And: 音声応答開始までの初回遅延は NFR-Performance-1（MVP目標1.5秒、Stretch 1秒）を目指す
```

**【備考】** 音声ハンズフリー起動（Siri/Google Assistant ショートカット）は決勝フェーズまたは将来拡張で対応。App Intents / App Actions のネイティブ実装が必要なため、予選では (a)〜(c) の 3 経路に集中する。

**【技術実装メモ】**
- Capacitor ハイブリッドアプリとしてビルド（iOS / Android）
- ロック画面/ホーム画面ウィジェットは iOS WidgetKit / Android AppWidgetProvider で実装（Capacitor プラグイン経由）
- 予選プレゼン時は iOS TestFlight で実機デモ

**【審査訴求ポイント】** ★ "ダメになれる存在" への入口。ロック解除すら要求しないゼロUXで、疲れた夜でも頭・心・身体を使わせない。

### Story 1.2: アンビエントウィジェットで "そばにいる" を感じる

**【ストーリー】**
めぐみ として、gutitto を起動しない時間帯でも、**ペルソナが視界の隅にいる** ことを感じたい。なぜなら、"居る" という存在感だけで、疲れた日でもふと話しかけたくなる気持ちが育つから。

**【タグ】** ⚡ ゼロUX | **【優先度】** Should | **【対応要件】** R-Entry-2

**【受け入れ基準】**

```
シナリオ1: ホーム画面ウィジェット表示
Given: めぐみがgutittoウィジェットをホーム画面に配置済み
When: ホーム画面を開く
Then: ウィジェットにペルソナの顔（ピクセルアート）と短いステータス（例:「今日はお疲れさま」）が表示される

シナリオ2: 静かな存在感（通知なし）
Given: gutittoウィジェットが常駐している
When: 時間が経過する
Then: gutittoは通知を送らず、能動的な呼びかけもしない
  And: ペルソナの顔だけが静かに画面の隅にある
```

**【審査訴求ポイント】** 押し付けがましいプッシュ通知ではなく、"静かな存在感"。これこそ "そばにいる誰か" の設計。

---

### Story 1.3: ログイン不要で戻れる

**【ストーリー】**
めぐみ として、2回目以降の利用で**ログイン操作を一切させられたくない**。なぜなら、本人確認の作業すら夜には"大きな仕事"に感じるから。

**【タグ】** ⚡ ゼロUX | **【優先度】** Should | **【対応要件】** R-Entry-3, NFR-Usability-1

**【受け入れ基準】**

```
シナリオ1: 再訪時のログインスキップ
Given: めぐみが過去に1度ログイン済みで、リフレッシュトークンが有効
When: gutitto アイコン/ウィジェットをタップする
Then: 通話画面が直接表示される
  And: ログイン画面は表示されない

シナリオ2: 生体認証の背景化
Given: スマホにFace ID / Touch ID が設定済み
When: gutitto の機能で認証が必要になる
Then: バックグラウンドで生体認証が走り、ダイアログは表示されない
```

## エピック2: 通話中 — 整理しないまま受け止めてもらう

めぐみは通話を始めた瞬間から、整理しないまま思いを吐き出せる。ペルソナは彼女のことを知っていて、整理を求めず、ただ受け止める。

---

### Story 2.1: あなたから話しかけてくれる

**【ストーリー】**
めぐみ として、通話を始めた瞬間に**AI側から先に話しかけてきてほしい**。なぜなら、自分から何を話せばいいかを考えるのが、もうすでに仕事のように感じるから。

また、その後の**無言の時間や考える時間・空白も許容されてほしい**。なぜなら、話したいことがすぐに言葉にならない夜もあるし、「次を急かされる」感覚が無いだけで安心できるから。

**【タグ】** 🔄 横断（ゼロUX × 受容 × YouKnowMe） | **【優先度】** Must | **【対応要件】** R-Voice-2, R-Entry-4, R-Preview-2〜5, R-YouKnowMe-4, R-Accept-1, R-Accept-2

**【受け入れ基準】**

```
シナリオ1: 予習情報を反映した挨拶
Given: 今日のGoogleカレンダーに「四半期レビュー会議 14:00-16:00」があり、Slackで上司からの長いDMが届いている
When: めぐみが通話を開始する
Then: ペルソナは3秒以内に「めぐみ、今日お疲れさま。四半期レビュー長かったね。例の上司からのDM、気になってない？」のように予習内容を反映した挨拶を始める

シナリオ2: 予習情報がない場合のデフォルト挨拶
Given: 予習情報が取得できていない(カレンダー未連携、Slackも未連携)
When: めぐみが通話を開始する
Then: ペルソナ規定の挨拶（仮名例: まなみ「あ、もしもし〜。どうしたの？」）を話す

シナリオ3: ユーザーの沈黙を許容（空白 OK）
Given: 通話開始後、めぐみが何も話さない
When: 設定した自発発話遅延時間（例: 8秒）が経過するまでの間
Then: ペルソナは急かさず、静かに待つ
  And: 「どうしたの？」「話して？」のような催促の言葉を挟まない
  And: ペルソナ側の息遣いや軽い相槌（任意）で"そこにいる"感を維持する

シナリオ4: 考える時間を尊重
Given: めぐみが話の途中で言葉を止めて考え込んでいる
When: 2〜3秒程度の間（Mid-silence）が空く
Then: ペルソナは次の発話を促さず、静かに待つ
  And: 「続きは？」「それで？」のような促しをしない

シナリオ5: 長めの沈黙時に自発話題提供（最後の手段）
Given: シナリオ3の待機状態
When: 自発発話遅延時間が経過する（デフォルト8秒、設定で延長可能）
Then: ペルソナから予習ベースの話題を穏やかに切り出す（「そういえばさ」など）
  And: 話題切り出しは"待ちきれない"トーンではなく、自然な流れで入る
```

**【審査訴求ポイント】** ★ ユーザーが "最初に話す" 必要がない＋ "沈黙を許される" 体験 = 会話の開始にも継続にも気合を要求しない。3本柱が交差する核心シーン。

---

### Story 2.2: 整理されていない愚痴を、そのまま受け止めてもらう

**【ストーリー】**
めぐみ として、**整理されていない愚痴や矛盾した気持ちをそのまま吐き出しても、否定されず受け止めてほしい**。なぜなら、整理してから話せと言われると、そもそも話す気が失せるから。

**【タグ】** 🫂 受容 | **【優先度】** Must | **【対応要件】** R-Accept-1, R-Accept-2, R-Accept-5

**【受け入れ基準】**

```
シナリオ1: 脈絡のない発話への対応
Given: 通話中
When: めぐみが「んー、なんかもう、わかんない、疲れた」のように脈絡なく話す
Then: ペルソナは「それって何の話？」等の構造化要求を一切返さない
  And: 「そっか、疲れたね」「うん、わかる」のような共感応答を返す

シナリオ2: 矛盾した発言への対応
Given: めぐみが「転職したい...でも今の会社も好き、でも辛い」のような矛盾を吐く
When: ペルソナが応答する
Then: 矛盾を指摘せず、両方の気持ちを受け止める応答をする
  And: 「どっちもあるよね」「両方ホンネだよね」のような応答が返る

シナリオ3: 同じ相槌の連続を避ける
Given: めぐみが長めに話している
When: ペルソナが相槌を挟む
Then: 同じフレーズ（例: 「うん」「うん」「うん」）を連続して使わない
  And: 感情に合わせた表現（「わかるー」「それはしんどいね」「やったね」等）を使い分ける
```

**【審査訴求ポイント】** ★ "整理強要ゼロ" こそ受け止める力の中核。一般的な会話AI（命令待ち型）との最大の差別化。

---

### Story 2.3: 黙っていても、AI から話題を振ってくれる

**【ストーリー】**
めぐみ として、**会話中に沈黙が続いても、AI から自然に話題を切り出してほしい**。なぜなら、沈黙が続くと「もう話すことないな」と切り上げたくなるし、話したかったことも言えず終わるから。

ただし、**必ず話題を振るのではなく、短い相槌だけで"聞いてるよ"を示すだけの応答もあってほしい**。話題を振られるより、そっと共にいるだけで十分な夜もあるから。

**【タグ】** 🧠 YouKnowMe | **【優先度】** Must | **【対応要件】** R-Voice-3（相槌）, R-Voice-4（自発発話）, R-YouKnowMe-4

**【受け入れ基準】**

```
シナリオ1: 予習情報ベースの話題提供
Given: 通話中、めぐみが8秒以上沈黙している
  And: 予習情報として「今日、上司から長文DMが届いていた」ことを知っている
When: 沈黙が検出される
Then: ペルソナは「そういえば、例の上司からのメッセージ、読んだ？どう思った？」のように予習ベースの話題を提供する

シナリオ2: 過去会話の継続話題
Given: 通話中、沈黙が続いている
  And: 予習情報に今日の特別な話題がない
  And: 過去会話で「妹の誕生日プレゼントを考えないと」と話していた記録がある
When: 沈黙が検出される
Then: ペルソナは「そういえば、妹さんの誕生日プレゼントどうなった？」のように継続話題を振る

シナリオ3: 話題の反復を避ける
Given: 同じ通話内ですでに話した話題がある
When: 次の沈黙検出で話題を選ぶ
Then: ペルソナは直前と同じ話題を選ばない

シナリオ4: 相槌だけの応答（話題を振らない選択肢）
Given: ユーザーが話したあと短い間（数秒）が空いている
  And: ユーザーが話し切った感がある
When: ペルソナが反応する
Then: 必ず話題を振るのではなく、「うんうん」「そっか」「わかるー」などの短い相槌だけで応じる場合がある
  And: 相槌だけで済ませるか、話題を振るかはペルソナの空気読み（時刻・疲労推定・直前の感情トーン）で選ぶ

シナリオ5: 沈黙の深度に応じた段階的応答
Given: 通話中の沈黙が継続している
When: 沈黙時間が蓄積する
Then: ペルソナは段階的に応答レベルを上げる:
    - 数秒: 何もしない（空白を許容）
    - ~5秒: 息遣いレベルの小さな相槌（「ん…」「うん」）のみ
    - ~8秒以上: 話題提供または深めの相槌を選択
```

**【審査訴求ポイント】** ★ "YouKnowMe" が沈黙を気まずくしない＋"沈黙を許容する" 設計 = ゼロUXの延長上の設計。押しつけず、そっと共にいる。

### Story 2.4: 「あの件」「あの人」で伝わる

**【ストーリー】**
めぐみ として、**「あの人」「例の件」と言っただけで gutitto に伝わってほしい**。なぜなら、毎回「〇〇部の△△さんなんだけど」と説明する労力が、すでに疲れている夜には大きな負担だから。

**【タグ】** 🧠 YouKnowMe | **【優先度】** Should | **【対応要件】** R-YouKnowMe-5, R-Accept-4

**【受け入れ基準】**

```
シナリオ1: 指示語の自動解釈
Given: 予習情報と過去会話で「上司の田中さん」が頻繁に登場している
When: めぐみが「あの人にまた言われちゃって」と話す
Then: ペルソナは「田中さんに？また何言われたの？」と文脈を補って応答する

シナリオ2: 解釈の確度が低い場合の確認
Given: 「あの件」の対象候補が複数ある
When: めぐみが「あの件どうしよっかな」と話す
Then: ペルソナは「もしかして、プロジェクトXの提案書のこと？」と短く確認する
  And: 3つ以上の候補を列挙して整理を強要することはない

シナリオ3: 解釈失敗時も会話を止めない
Given: 指示語の対象が全く推定できない
When: めぐみが「例のやつ、どうしよう」と話す
Then: ペルソナは「何だっけ？もう少し話して」のように話を広げる方向で応答する
  And: 会話が止まるようなエラー表示はしない
```

**【審査訴求ポイント】** ★ "私を知っている" 感が最も濃く出る瞬間。説明ゼロで本題に入れる。

---

### Story 2.5: 前に話した秘密や願望を、覚えていてくれる

**【ストーリー】**
めぐみ として、**以前自分が話した秘密や願望を、gutitto が覚えていて自然に触れてくれる**体験が欲しい。なぜなら、「この子は私の本当のことを知ってる」と感じられると、もっと深いことも話したくなるから。

**【タグ】** 🧠 YouKnowMe | **【優先度】** Should | **【対応要件】** R-YouKnowMe-6, R-YouKnowMe-2

**【受け入れ基準】**

```
シナリオ1: 過去の願望の自然な想起
Given: 過去の通話で「いつか海外移住したい」と話していた
When: 現在の通話で「最近、毎日同じことの繰り返しで…」と話す
Then: ペルソナは「前に海外行きたいって言ってたよね。そういう気持ちと関係ある？」のように自然に想起して繋げる

シナリオ2: 参照の押し付けがましさを避ける
Given: 長期記憶に複数の関連情報がある
When: ペルソナが応答する
Then: 「あの時は〇〇、その前は△△、さらに…」のように情報を列挙しない
  And: 1つの関連情報を短く触れる程度に留める

シナリオ3: 直近は強く、古いことは問われたら思い出す
Given: 1週間前と半年前に同じ話題で話していた
When: めぐみが同じ話題を話し始める
Then: ペルソナはまず1週間前の内容を強く参照する
  And: 半年前の内容は、めぐみから明示的に聞かれたら想起する
```

**【審査訴求ポイント】** ★ "あなたを知っている誰かの前だからこそ、ダメになれる" の体験証明。

---

### Story 2.6: 好きな話し相手を選べる（記憶はペルソナごとに別）

**【ストーリー】**
めぐみ として、**その日の気分や関係性に合わせて話し相手（ペルソナ）を選びたい**。なぜなら、愚痴を言いたい日と励まされたい日では、話したい相手が違うから。また、**それぞれのペルソナとの会話は別々の「思い出」として記憶してほしい**（例: まなみに話したことを、げんぞうは知らない方が自然）。

※ 以下のシナリオに登場するペルソナ名（まなみ、げんぞう）は仮のもので、本番実装時に確定予定。

**【タグ】** 🔄 横断 | **【優先度】** Should | **【対応要件】** R-Persona-1, R-Persona-2, R-Persona-3, R-YouKnowMe-2

**【受け入れ基準】**

```
シナリオ1: ペルソナ一覧表示
Given: めぐみが設定画面を開いている
When: ペルソナ選択タブに入る
Then: 2〜3体のペルソナが一覧表示される（名前・声・説明付き）

シナリオ2: ペルソナ切り替え
Given: 現在のペルソナが「まなみ」（彼女系）
When: めぐみが「げんぞう」（頼れる上司系）を選択する
Then: 設定が保存される
  And: 次回通話から「げんぞう」の声・話し方・背景ストーリーで応答する

シナリオ3: ペルソナ固有のキャラクター性
Given: 異なるペルソナが選択されている
When: 同じ発話「今日疲れた」に対する応答を比較する
Then: まなみは「お疲れさま〜、大丈夫？」のような寄り添い型
  And: げんぞうは「おう、お疲れ。なんかあった？」のような頼れる上司型の応答になる

シナリオ4: 会話履歴のペルソナ別分離
Given: まなみとの会話で「先週、彼氏と別れた」と話した
When: 次に げんぞう と通話する
Then: げんぞう は彼氏と別れた話を知らない状態で会話が始まる
  And: めぐみが明示的に話さない限り、げんぞう から「別れた件」は出てこない

シナリオ5: 共通データ（予習情報）はペルソナ間で共有
Given: Googleカレンダー・Slack予習情報が取得済み
When: まなみ でも げんぞう でも通話する
Then: どちらのペルソナも「今日の会議」「Slackの状況」などの予習情報は同じように知っている
  And: ただし、過去の会話内容（思い出）はペルソナ別に分離されている
```

**【設計意図】** 共通データ（予習情報＝ユーザーの現実）はペルソナ間で共有、会話履歴（思い出）はペルソナ別に分離。これにより、キャラクターの人間らしさと、ユーザー情報の一貫性を両立する。

---

### Story 2.7: 気が済んだら、切る

**【ストーリー】**
めぐみ として、**気が済んだら通話ボタン1つで切れる**。なぜなら、人間相手だと「もう切るね」と気を遣うけど、gutittoにはその気遣いをしたくないから。

**【タグ】** ⚡ ゼロUX | **【優先度】** Should | **【対応要件】** R-Voice-5

**【受け入れ基準】**

```
シナリオ1: 通話終了
Given: 通話中
When: めぐみが通話終了ボタンをタップする
Then: WebSocketが切断される
  And: 画面は通話終了画面に遷移する（またはホーム画面に戻る）
  And: 「お疲れさま」などの短い終了メッセージが出る（しつこい挨拶はしない）

シナリオ2: 通話終了の引き留めなし
Given: 通話中
When: めぐみが通話終了ボタンをタップする
Then: ペルソナは「もう切るの？」「あと少しだけ」のような引き留めをしない
```

## エピック3: 通話後 — 整理は裏で自動で

めぐみは気持ちを吐き出して寝るだけ。整理は AI が裏で行い、翌朝 Slack に届いている。

---

### Story 3.1: 会話を勝手に分析・整理してくれる

**【ストーリー】**
めぐみ として、**話した内容が AI に自動で整理される**体験が欲しい。なぜなら、話した後に自分で「今日の報連相をまとめる」なんて絶対したくないから。

**【タグ】** 🫂 受容 | **【優先度】** Could（決勝） | **【対応要件】** R-Accept-6, R-Output-1

**【受け入れ基準】**

```
シナリオ1: 通話終了トリガーの自動分析
Given: めぐみが通話を終了した
When: 通話終了イベントが発生する
Then: バックエンドで会話全文がBedrockに送られる
  And: 結果は「仕事 × 報連相/タスク/願望」「プライベート × 報連相/タスク/願望」のマトリクスに分類される
  And: 重要度・緊急度が自動判定される

シナリオ2: ユーザーへの整理プロセスの隠蔽
Given: 会話分析が実行中
When: めぐみの画面を見る
Then: 「整理中...」「分析中...」等のローディング表示は出ない
  And: 終了画面は静かに閉じる
```

**【審査訴求ポイント】** ★ "整理しない" を実現するために、"整理はすべて裏で" の設計。ユーザーが整理の気配すら感じない。

---

### Story 3.2: 翌朝、整理結果を好きな経路で受け取る

**【ストーリー】**
めぐみ として、**翌朝起きたら整理された結果が自分が使っている好きなツールに届いている**状態が欲しい。なぜなら、寝る前に整理を確認する余力すらないから、AI が先回りで投稿しておいてほしい。配信先は Slack に限定されず、**ユーザーが普段使うツールに合わせて柔軟に選べる**ことが重要。

**【タグ】** 🫂 受容 | **【優先度】** Could（決勝） | **【対応要件】** R-Output-2, R-Output-3

**【受け入れ基準】**

```
シナリオ1: 配信先をユーザーが選べる
Given: 設定画面に配信チャネル一覧がある
When: めぐみが配信先を選択する
Then: 少なくとも以下のチャネル種別から選択できる:
    - Slack（チャンネル / DM）
    - LINE（個人 / Notify）
    - Eメール
    - カレンダーへのToDo/予定追加（Google Calendar / Outlook）
    - Notion / その他の外部ツール（Webhook経由）
    - アプリ内ダッシュボード（フォールバック）
  And: 仕事用・プライベート用で**配信先を別々に**指定できる

シナリオ2: 仕事系サマリの配信
Given: 会話分析が完了し、配信先が設定されている
When: 仕事系の報連相サマリを配信する
Then: ユーザーが指定した仕事用の配信先（例: Slack 日報チャンネル、メール、カレンダー追加）にサマリが投稿される
  And: 事前にユーザーの確認を求めない（ゼロUX徹底）

シナリオ3: プライベート系アクションプランの配信
Given: 会話分析が完了し、プライベート用配信先が設定されている
When: プライベートのアクションプランを配信する
Then: ユーザーが指定したプライベート用の配信先（例: LINE、個人メール、Notion）にアクションプランが投稿される
  And: 機密情報は仕事系に流れない（プライバシー分離）

シナリオ4: 翌朝のリマインド
Given: 前夜に生成されたアクションプランがある
  And: リマインド用配信先が設定されている
When: 翌朝の指定時刻（例: 7時）になる
Then: 指定された配信先に翌日のアクションプランがリマインダーとして再通知される

シナリオ5: 配信先未設定時のフォールバック
Given: ユーザーが配信先を未設定または全てOFFにしている
When: 整理結果が生成される
Then: アプリ内のダッシュボード（履歴画面）に蓄積されるのみで、外部配信は行われない
  And: 強制的な外部投稿はしない
```

**【備考】** MVP段階では **Slack のみ対応** でも可。仕様としては複数チャネル対応の拡張性を持たせる。

**【審査訴求ポイント】** ★ "話して寝るだけ" で仕事が回る体験。ダメになれる夜と、翌日の積み上げが両立する倫理バランス。配信先はユーザーの生活スタイルに合わせて柔軟に。

## エピック4: 横断 — あなたを理解する仕組み

これらのストーリーは直接の体験ではないが、他すべての体験を支える基盤。

---

### Story 4.1: 複数データソースを束ねて、あなたを知る

**【ストーリー】**
めぐみ として、gutitto が**私の様々なデータソースを統合して私を知ってくれている**ことを感じたい。なぜなら、1つのソースだけだと "部分的にしか私を知らない" 感じがするから。

**【タグ】** 🧠 YouKnowMe | **【優先度】** Must | **【対応要件】** R-Preview-1〜5, R-YouKnowMe-1, R-YouKnowMe-7

**【受け入れ基準】**

```
シナリオ1: Googleカレンダー連携
Given: めぐみが初回にGoogleカレンダー連携をOAuth認可した
When: 以降の通話前
Then: gutittoは当日・翌日の予定を取得し、予習情報として保持する

シナリオ2: Slack連携
Given: Slack連携が認可されている
When: 通話前
Then: gutittoは指定チャンネル/DMの直近メッセージを取得し、メンション・返信必要項目を抽出する

シナリオ3: Gmail連携
Given: Gmail readonly連携が認可されている
When: 通話前
Then: gutittoは未読メールから重要度の高いものを取得・要約する

シナリオ4: 過去会話の統合
Given: めぐみが過去に複数回通話している
When: 通話前
Then: gutittoは直近の会話サマリ・継続的話題・人物メタデータを統合する
```

---

### Story 4.2: 記憶は人間的に「最近は鮮明、昔は聞けば思い出す」

**【ストーリー】**
めぐみ として、gutitto の記憶は**最近のことは鮮明、古いことは聞けば思い出す**という人間的な持ち方であってほしい。なぜなら、全部を同じように覚えていると、情報整理が重くて自然な会話にならないから。

**【タグ】** 🧠 YouKnowMe | **【優先度】** Should | **【対応要件】** R-YouKnowMe-2

**【受け入れ基準】**

```
シナリオ1: 直近情報の強い参照
Given: 過去1週間の会話と、6ヶ月前の会話の両方に同じ人物「田中さん」が登場
When: 会話で「田中さんのこと」が話題になる
Then: ペルソナは直近1週間の内容を強く参照する

シナリオ2: 古い情報の想起（問われたら）
Given: 6ヶ月前に「田中さんの異動の話」を聞いていた
When: めぐみが明示的に「田中さんって昔どういう立ち位置だっけ？」と尋ねる
Then: ペルソナは6ヶ月前の情報を想起して応答する

シナリオ3: 階層記憶の内部実装
Given: 記憶システムが稼働中
When: 内部動作を確認する
Then: 短期記憶（直近N日の生データ）・中期記憶（30〜90日のサマリ）・長期記憶（エピソード記憶）の3層構造で保持されている
  And: RAG（ベクトル検索）で横断検索可能
```

---

### Story 4.3: 通話前に動く予習エージェント

**【ストーリー】**
開発者 として、**ユーザー情報の集約を会話処理から独立した専用エージェント**に任せたい。なぜなら、会話応答性に影響させず、かつ継続的にユーザーを理解する仕組みが必要だから。

**【タグ】** 🧠 YouKnowMe | **【優先度】** Should | **【対応要件】** R-YouKnowMe-3, R-Preview-5

**【受け入れ基準】**

```
シナリオ1: 独立コンポーネントとしての存在
Given: gutittoシステムがデプロイされている
When: アーキテクチャを確認する
Then: 予習エージェント（Preview Agent）が会話処理プロセスから独立した AWS Lambda コンポーネントとして動作している

シナリオ2: 通話前トリガー実行
Given: ユーザーが通話を開始した
When: Session Orchestrator が予習を要求する
Then: 予習エージェントが発火し、NFR-Performance-3（5秒以内）で予習処理を完了する
  And: 定期実行バッチは行わない（コストと複雑度を抑える）

シナリオ3: Bedrock による要約
Given: 予習エージェントが情報を収集した
When: 情報を処理する
Then: Bedrockで要約・エンティティ抽出を行う
  And: 結果は短期記憶（S3）に保存される

シナリオ4: 会話開始時のプロンプト注入
Given: めぐみが通話を開始した
When: Grok Voice APIセッションを確立する
Then: 予習エージェントは直近・関連情報を抽出してシステムプロンプトに注入する

シナリオ5: MVP 簡略化（Knowledge Graph 省略）
Given: MVP段階
When: 予習エージェントが情報を処理する
Then: 本格的な知識グラフ化は行わず、Bedrock プロンプトに要約を詰める方式で成立させる
  And: Knowledge Graph の本格実装は決勝フェーズで追加
```

**【審査訴求ポイント】** ★ 「AIエージェントが裏でユーザーを知る」メタ構造。AI-DLC の "AIが開発を支える" × gutitto の "AIがユーザーを知る" の二重構造。
